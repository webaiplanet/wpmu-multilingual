# WPMU 多语言插件 0.9.8.3 文章关联改进与测试汇报

## 1. 汇报结论

- 插件当前版本：`0.9.8.3`。
- 部署状态：WordPress Multisite 网络启用。
- 源站：Blog ID `2`。
- 目标站：13 个。
- 本轮目标：阻止错误文章关系导致的覆盖、翻译写错、错误删除、错误链接和 slug 冲突。
- 代码改进、数据库唯一约束、PHP 语法检查和生产数据只读审计已经完成。
- 真实文章同步、翻译、回收站、恢复和永久删除尚未执行，必须由测试人员使用专用测试文章完成灰度测试。
- 禁止对生产关系执行旧版重建；旧重建入口已经停用。

## 2. 原有主要风险

改进前存在以下高风险行为：

1. 没有明确关系时，可能把目标站相同 ID 的文章认作翻译文章。
2. 相同 ID 未命中时，可能把相同 slug 的文章认作翻译文章。
3. 同步、翻译、删除、恢复和前台链接没有共用的目标身份校验。
4. 永久删除策略为 `all` 时，错误关系可能删除无关目标文章。
5. 旧版“重建关联”会先清空关系表，再使用 meta、ID 和 slug 猜测关系。
6. 数据库没有阻止一个目标文章同时属于多个源文章。
7. 强制同步 slug 时直接更新 `posts.post_name`，可能制造重复或歧义 URL。

## 3. 已完成的改进

### 3.1 文章身份统一校验

新增并集中使用文章关系、目标占用和目标身份校验接口，校验内容包括：

- 关系表中的源站、源文章、目标站和目标文章是否完整。
- 关系表指向的目标文章是否真实存在。
- 目标文章类型是否与关系记录一致。
- 目标文章来源 meta 是否与当前源文章一致。
- 目标文章是否已经被其他源文章关系占用。
- 目标文章是否在目标站被人工回收或删除。

历史关系如果同时缺少两个来源 meta，但关系表本身没有冲突，继续按 `legacy_relation` 兼容，不批量重建或改号。历史关系在下一次安全写入时才补写：

```text
_wpmu_ml_source_blog_id
_wpmu_ml_source_post_id
_wpmu_ml_source_lang
_wpmu_ml_target_lang
_wpmu_ml_relation_version=2
```

### 3.2 新文章不再按 ID 或 slug 猜测关系

新文章首次同步现在只允许以下流程：

```text
查询已有关系
→ 校验已有关系
→ 按完整来源 meta 查找
→ 没有候选时创建新目标文章
→ 使用 wp_insert_post() 返回的真实目标 ID
→ 写入来源 meta
→ 写入关系表
→ 创建翻译任务
```

已经删除生产运行中的以下自动兜底：

```text
相同 ID 自动认领
相同 slug 自动认领
```

### 3.3 写入和删除路径保护

以下操作执行前均校验目标身份：

- 源文章自动同步和更新同步。
- 翻译队列处理。
- OpenAI 兼容翻译写回。
- OpenCC 写回。
- 外部 Agent Payload 和 Result 写回。
- Agent Tools 跨站文章写回。
- 批量文章状态和关系状态修改。
- slug 单篇修复和批量修复。
- 源站回收站、恢复和永久删除同步。
- 目标站人工回收、恢复和永久删除事件。

身份异常时停止操作并记录错误，不继续覆盖、翻译或删除目标文章。

### 3.4 slug 冲突保护

`0.9.8.3` 新增统一 slug 可用性检查：

- 同类型文章已经占用源 slug 时阻断写入。
- 附件与文章发生相关 slug 冲突时阻断写入。
- 忽略已经位于 `trash` 或 `auto-draft` 的对象。
- 不认领现有同 slug 文章。
- 不覆盖现有同 slug 文章。
- 不自动为源 slug 添加 `-2` 后继续建立错误关系。
- 冲突错误码统一为 `target_slug_conflict`。

检查已经接入同步、队列、OpenAI、OpenCC、Agent、Agent Tools 和 slug 修复。强制 slug 写入内部还会二次检查，调用方必须处理错误。

Agent Result 原来的独立数据库 slug 写入已经删除，改用统一受保护入口。

### 3.5 数据库唯一约束

文章关系表已增加：

```sql
UNIQUE KEY target_unique (target_blog_id, target_post_id)
```

该约束保证同一个目标站文章不能被多个源文章关系同时占用。关系保存也已从危险的 `REPLACE` 改为明确的 `UPDATE` 或 `INSERT`，唯一键冲突只会失败，不会替换并删除旧关系。

数据库迁移前备份：

```text
/home/www/likacloud-backups/wpmu-ml-post-relations-before-0.9.8.2-20260721.sql
```

备份 SHA-256：

```text

```

### 3.6 危险重建停用

- 后台“重建关联”按钮已经移除。
- 后台直接请求重建会拒绝执行。
- `wp wpmu-ml rebuild` 会明确拒绝执行。
- 运行代码中的 `TRUNCATE`、同 ID 猜测和同 slug 猜测重建已经删除。

新增严格、非破坏性的关系恢复命令：

```bash
wp wpmu-ml reconcile-relations --target_blog_id=1 --limit=500 --allow-root --skip-themes
```

该命令默认 dry-run，只接受完整来源 meta 作为依据，不使用 ID 或 slug 猜测。禁止在未审核候选前增加 `--apply`。

### 3.7 前台链接和 hreflang 保护

- 语言切换器只使用通过身份校验的关系。
- 后台当前页语言链接只使用有效关系。
- `hreflang` 不输出缺失或身份异常的目标。
- `hreflang` 排除未发布、密码保护、站点禁止索引及常见 SEO 插件明确设置 noindex 的页面。
- 新增过滤器 `wpmu_ml_post_is_indexable_for_hreflang`。

当前站点 1 和站点 2 的 `blog_public=0`，因此当前不输出 `hreflang` 属于预期行为；语言切换功能仍应可用。

### 3.8 审计能力

详细只读审计：

```bash
wp wpmu-ml audit-relations --target_blog_id=1 --limit=500 --offset=0 --allow-root --skip-themes
```

全站汇总只读审计：

```bash
wp wpmu-ml audit-relations --summary --allow-root --skip-themes
```

审计现在覆盖：缺失源文章、缺失目标文章、类型冲突、身份 meta、重复目标关系、异常关系状态和 slug 冲突。审计命令不修改数据库。

## 6. 人工测试详细步骤

### T01 版本检查

```bash
wp plugin get wpmu-multilingual --field=version --allow-root --skip-themes
wp plugin status wpmu-multilingual --allow-root --skip-themes
wp eval 'echo WPMU_ML_VERSION . "|" . WPMU_Multilingual::VERSION;' --allow-root --skip-themes
```

预期：版本均为 `0.9.8.3`，状态为 `Network Active`。

### T02 审计基线

```bash
wp wpmu-ml audit-relations --summary --allow-root --skip-themes
```

测试前后均执行一次。预期除测试文章产生的明确变化外，关系总量、重复目标关系和身份冲突不应出现异常增长。

### T03 正常新文章同步

1. 在源站创建一篇专用草稿，标题带明显测试标记。
2. 使用全网唯一 slug，例如 `wpmu-ml-test-日期-随机值`。
3. 触发正常同步。
4. 检查目标文章、来源 meta、关系表和翻译任务。

预期：

- 不按源文章 ID 强制创建目标 ID。
- 关系表记录 `wp_insert_post()` 返回的实际目标 ID。
- 目标只属于当前源文章。
- 来源 meta 与关系表一致。
- 没有覆盖目标站其他文章。

### T04 相同 ID 占用

该场景需要精确控制目标数据库 ID，只允许在测试环境构造。

预期：相同数字 ID 不能作为认领依据。只要没有关系和来源 meta，插件必须创建新的目标文章并记录实际返回 ID。

### T05 相同 slug 占用

1. 在目标测试站创建一篇无关专用草稿，占用测试 slug。
2. 在源站创建另一篇测试文章，使用相同 slug 和相同 post type。
3. 触发同步或翻译。
4. 记录错误日志和两个测试文章的内容、ID、slug、状态。

预期：

- 返回或记录 `target_slug_conflict`。
- 已占用 slug 的文章 ID、标题、正文、状态和 meta 均不变化。
- 不把占用文章认作翻译文章。
- 不创建指向占用文章的关系。
- 不创建错误翻译任务。

### T06 正常更新与翻译保护

1. 使用 T03 建立的正常测试关系。
2. 修改源文章标题、正文和摘要。
3. 分别测试未翻译目标和已翻译目标。

预期：只更新明确关系目标；启用译文保护时，已翻译正文不被源文直接覆盖，而是标记为需要更新。

### T07-T08 翻译写回

分别执行实际启用的 OpenAI、OpenCC、Agent 和 Agent Tools 测试。

预期：

- 写回前重新校验关系身份和 slug。
- 写回目标 ID 与关系表一致。
- 目标 slug 保持源站 slug，不使用译文标题生成 slug。
- 冲突任务停止并给出明确错误。
- 不修改无关文章。

### T09-T11 生命周期测试

使用可丢弃测试文章依次测试：

```text
源站移入回收站
→ 源站恢复
→ 目标站人工移入回收站
→ 目标站人工恢复
```

永久删除只在测试环境执行。每一步记录源/目标文章状态、关系状态、任务状态和日志。

预期：源站操作只处理对应目标；目标站人工操作会停止自动写入并留下可审计状态。

### T12-T13 前台测试

检查文章页语言切换链接：

- 链接必须指向正确关系目标。
- 缺失和身份冲突关系不能生成错误链接。
- 未发布目标按现有配置隐藏。

现存 slug 冲突文章可能共享同一个前台 URL，写入保护不会自动改变这些历史 URL。测试人员应记录这类链接，不应通过自动认领或直接改关系处理。

检查 `hreflang` 时必须使用允许搜索引擎索引的测试站。当前 `blog_public=0` 的站点不输出 `hreflang` 是正确结果。

### T14 重建保护

```bash
wp wpmu-ml rebuild --allow-root --skip-themes
```

预期：命令明确拒绝执行，关系表总数不变。不要尝试绕过保护。

### T15 并发唯一性

仅在测试环境同时尝试让两个源文章占用同一目标文章。

预期：数据库唯一键阻止第二条关系；已有关系不被 `REPLACE` 删除或替换。

## 7. 测试通过标准

全部满足以下条件才可确认灰度通过：

1. 无任何无关目标文章被更新、翻译、回收或删除。
2. 新关系不依赖相同 ID 或相同 slug 猜测。
3. slug 冲突稳定返回 `target_slug_conflict`，占用文章保持不变。
4. 正常同步和全部启用的翻译引擎可写入正确目标。
5. 源站与目标站生命周期操作符合配置，异常目标停止自动写入。
6. 非 slug 冲突文章的语言切换链接正确，缺失或身份异常关系不输出链接。
7. 数据库没有重复目标关系，关系总数只产生预期测试增量。
8. PHP 错误日志没有新增 fatal error、未捕获异常或数据库唯一键替换异常。

## 8. 已知历史数据问题

以下为现存数据，不是本轮代码自动制造：

- 130 条关系的源文章缺失，同时对应 130 个目标缺失，每个目标站 10 条。
- 149,474 条历史关系缺少来源身份 meta，当前按历史关系兼容策略继续使用。
- 234 条关系存在目标 slug 占用冲突，每个目标站 18 条。
- 当前没有源/目标 post type 冲突。
- 当前没有来源身份 meta 冲突。
- 当前没有重复目标关系。

这些历史问题不得通过 ID、slug 猜测或批量重建自动修复。slug 冲突需要逐组确认哪篇文章保留哪个 URL，再制定单独迁移方案；在解决前，相关语言切换和 `hreflang` URL 也可能存在歧义。
